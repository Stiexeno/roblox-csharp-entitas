---

Folder structure is feature-based and Rojo-friendly. Files ending in `.server.cs` become Server Scripts; `.client.cs` become Local Scripts; plain `.cs` become ModuleScripts.

```
src/
+-- Common/                    # Shared components and utilities
|   +-- CommonComponents.cs    # Cross-feature components (Id, OwnerId, Dead, Ready, etc.)
|   +-- Extensions/            # Extension methods
|   +-- Utilities/             # Stateless helpers
|
+-- Gameplay/                  # All gameplay features
|   +-- Combat/                # One folder per feature
|   |   +-- Configs/           # Static / DI-bound config data (plain C# classes)
|   |   +-- Services/          # Domain services + Factories for this feature
|   |   +-- Systems/           # IExecuteSystem / IInitializeSystem / etc.
|   |   +-- CombatComponents.cs
|   |   +-- CombatFeature.cs   # Pipeline composition for this feature
|   +-- Player/
|   |   +-- ...
|   +-- ...
|   +-- GameplayCoreFeature.cs # Composes every gameplay Feature in order
|
+-- Infrastructure/            # Core infrastructure modules
|   +-- ConfigsManagement/     # Config loading (e.g. from MemoryStore / DataStore)
|   +-- Installers/            # DI installers (ServerInstaller / ClientInstaller / shared)
|   +-- StateManagement/       # Game state machine and states
|   +-- Bootstrap.server.cs    # Server entry point
|   +-- Bootstrap.client.cs    # Client entry point
|   +-- Bootstrap.shared.cs    # Shared init (Contexts.sharedInstance, etc.)
|
+-- UI/                        # UI scripts (Razor/Blazor if you go that path)
|
+-- Storage/                   # DataStore wrappers and snapshots
|
+-- Generated/                 # Auto-generated by the entities plugin — DO NOT EDIT
```

Rules:

- **Feature folder names** are singular (`Combat`, `Player`, `Inventory`) — never `Combats`.
- **Class names** are singular too. `CombatFeature` / `CombatComponents` / `CombatFactory`.
- **One components file per feature** lives in the feature root: `{Feature}Components.cs`.
- **Systems** go into the feature's `Systems/` subfolder.
- **Factories + Services + Queries** go into `Services/` (they share the same place because they all wire via DI). If a feature has many services, split into `Services/Factories/`, `Services/Queries/` later.
- **Configs** are plain C# classes (no `ScriptableObject`). Static-data configs can be `public static class GameConfig` with const fields. Loadable configs are DI-bound classes that get their data from `MemoryStore` / a JSON `Resource` / etc.
- **Namespaces mirror folder paths.** `src/Gameplay/Combat/Systems/StartAttackSystem.cs` → `namespace MyGame.Gameplay.Combat.Systems`.

## Server / Client split convention

Use the file-name suffix to scope code to a runtime side:

- `Bootstrap.server.cs` — only runs on the server (Roblox Server Script)
- `Bootstrap.client.cs` — only runs on the client (Local Script)
- `Bootstrap.shared.cs` (or no suffix) — ModuleScript, replicated to both sides

For systems and factories that are inherently one-sided (e.g. `SpawnEnemySystem` only runs server-side), the `.server.cs` suffix on the file is the cleanest signal. Don't try to gate execution at runtime with `if (RunService.IsServer())` checks scattered through shared modules — split the file.
